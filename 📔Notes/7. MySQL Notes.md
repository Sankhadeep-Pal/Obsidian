## What is Database?
A **database** is a structured collection of data that is organized so it can be easily accessed, managed, and updated. Think of it as a digital filing system where information is stored in a way that software applications can quickly retrieve, add, modify, or delete data.

Key points:

- **Structured storage**: Data is organized in tables, rows, and columns (in relational databases) or in documents, key-value pairs, or graphs (in NoSQL databases).

- **Efficient retrieval**: Databases are optimized for searching, filtering, and aggregating data quickly.

- **Consistency and integrity**: They enforce rules so data remains accurate and reliable.

- **Multi-user access**: Multiple users or applications can read and write data concurrently without corruption.

- **Scalability**: A database can handle growth‚Äîmore data, more users, or more requests‚Äîwithout breaking or slowing down. It‚Äôs a critical factor in real-world systems because data and traffic almost always grow over time.

- **Security**: Database can protect your data from unauthorized access, corruption, or loss.
	
	 Some ways for database to achieve that:
	 
	1. **Authentication & Authorization**
	    
	    - Only allow the right people or applications to access the database.
	    
	    - Example: A user logs in (authentication), and the system only lets them see their own data (authorization).
	
	2. **Encryption**
	    
	    - **At rest**: Data stored on disk is encrypted so attackers can‚Äôt read it if they get the files.
	    
	    - **In transit**: Data traveling over the network is encrypted to prevent eavesdropping.
	
	3. **Access Control & Privileges**
	    
	    - Limit what users or apps can do. For example, a reporting tool might only be able to read data, not delete it.
	
	4. **Auditing & Monitoring**
	    
	    - Track who accessed the database and what they did. This helps detect breaches or misuse quickly.
	
	5. **Backups & Recovery**
	    
	    - Even with strong security, mistakes or attacks happen. Regular backups let you restore data without catastrophic loss.
	
	6. **SQL Injection & Input Validation**
	    
	    - Protect against attacks where malicious users try to trick the database into executing harmful commands.

**Example:** A social media app uses a database to store user profiles, posts, likes, and comments. Without a database, it would be nearly impossible to manage this data efficiently.

## CRUD Operations
CRUD operations are the **basic actions you can perform on data in a database**. Every database‚Äîrelational or NoSQL‚Äîsupports them in some form. The acronym **CRUD** stands for:

1. **Create** ‚Äì Add new data.
    
    - Example: Adding a new user to a social media app.
    
    - SQL example:
        
```sql
INSERT INTO users (name, email) VALUES ('Sankho', 'sankho@example.com');
```

2. **Read** ‚Äì Retrieve or view existing data.
    
    - Example: Fetching a user‚Äôs profile or post.
    
    - SQL example:
        
```sql
SELECT * FROM users WHERE name='Sankho';
```

3. **Update** ‚Äì Modify existing data.
    
    - Example: Changing a user‚Äôs email address.
    
    - SQL example:
        
```sql
UPDATE users SET email='newemail@example.com' WHERE name='Sankho';
```

4. **Delete** ‚Äì Remove data.
    
    - Example: Deleting a user account.
    
    - SQL example:
        
```sql
DELETE FROM users WHERE name='Sankho';
```


In essence, **CRUD is the foundation of all database interactions**. If you master CRUD, you can handle almost any database task and start thinking about more advanced stuff like indexing, transactions, and optimization.

## SQL
SQL (**Structured Query Language**) is a programming language designed specifically for **managing and manipulating relational databases (RDBMS)**.

- **Relational databases** store data in **tables** (rows = records, columns = fields).

- SQL allows you to:
    
    - Retrieve data (`SELECT`)
    
    - Add data (`INSERT`)
    
    - Update data (`UPDATE`)
    
    - Delete data (`DELETE`)
    
    - Control access (`GRANT` / `REVOKE`)
    
    - Define database structure (`CREATE`, `ALTER`, `DROP`)

Think of SQL as **the language your database understands**. Without it, you can‚Äôt tell your database what to do.

### üîë Key SQL Components
SQL commands are grouped into **categories**:

#### a) DDL (Data Definition Language)
Deals with **structure** of the database (tables, schemas, indexes).

- `CREATE` ‚Äì Make a new table or database
    
```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    email VARCHAR(100)
);
```

- `ALTER` ‚Äì Change table structure
    
```sql
ALTER TABLE users ADD COLUMN age INT;
```

- `DROP` ‚Äì Delete table or database
	
```sql
DROP TABLE users;
```

#### b) DML (Data Manipulation Language)
Deals with  **CRUD operations** on data.

- `INSERT` ‚Äì Add new records

- `SELECT` ‚Äì Read data

- `UPDATE` ‚Äì Modify existing data

- `DELETE` ‚Äì Remove data

**Example:**

```sql
INSERT INTO users (name, email) VALUES ('Sankho', 'sankho@example.com');
SELECT * FROM users WHERE name='Sankho';
UPDATE users SET email='new@example.com' WHERE name='Sankho';
DELETE FROM users WHERE id=1;
```

#### c) DCL (Data Control Language)
Deals with **permissions and access control**.

- `GRANT` ‚Äì Give permissions

- `REVOKE` ‚Äì Remove permissions

Example:

```sql
GRANT SELECT, INSERT ON users TO 'app_user';
REVOKE DELETE ON users FROM 'app_user';
```

#### d) TCL (Transaction Control Language)
Deals with **transactions**‚Äîa set of operations treated as a single unit.

- `COMMIT` ‚Äì Save changes permanently

- `ROLLBACK` ‚Äì Undo changes

- `SAVEPOINT` ‚Äì Set checkpoints within a transaction

Example:

```sql
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

This ensures either **all changes happen or none**‚Äîcritical for financial systems.

### ‚úçÔ∏è How SQL Works

1. **You write a query** (like `SELECT * FROM users`).

2. **Database engine parses it**‚Äîchecks syntax and execution plan.

3. **Query optimizer** decides the fastest way to fetch data.

4. **Data is returned** in a structured table format.

SQL is **declarative**, not procedural. You **describe what you want**, not how to do it. The database engine figures out the best way.

### üí™Why SQL is Still Essential

Even with NoSQL, SQL is **the backbone of modern applications** because:

- It‚Äôs standardized, widely supported, and proven.

- Skills transfer across MySQL, PostgreSQL, Oracle, SQL Server, etc.

- Understanding relational thinking helps you design **efficient, reliable systems**.

## SQL v/s NoSQL

### 1. Data Model / Structure

|Feature|SQL (Relational)|NoSQL (Non-Relational)|
|---|---|---|
|Structure|Tables with **rows & columns**|Flexible: documents (JSON), key-value pairs, graphs, or wide-columns|
|Schema|Fixed schema (must define structure in advance)|Schema-less or dynamic (fields can vary between records)|
|Example|MySQL, PostgreSQL, Oracle|MongoDB, Cassandra, Redis, Neo4j|

**Takeaway:** SQL is strict and predictable; NoSQL is flexible and adapts to changing data.

### 2. Query Language

|Feature|SQL|NoSQL|
|---|---|---|
|Querying|Uses **SQL** (Structured Query Language)|Uses **API queries**, specific to the database (e.g., MongoDB uses JSON-like queries)|
|Complexity|Handles **complex joins, filters, aggregations** easily|Less ideal for complex multi-table relationships; optimized for **simple, fast operations**|

**Takeaway:** SQL is great for analytics and complex reporting; NoSQL is optimized for **speed and scale**.

### 3. Scalability

|Feature|SQL|NoSQL|
|---|---|---|
|Scaling|Vertical (upgrade a single server: CPU, RAM)|Horizontal (add more servers/nodes, easy to scale out)|
|Performance|Can slow down with huge data if not optimized|Handles massive traffic and huge datasets efficiently|

**Takeaway:** SQL can struggle with massive distributed systems; NoSQL thrives in large-scale, high-traffic apps.

### 4. Transactions & Consistency

|Feature|SQL|NoSQL|
|---|---|---|
|ACID Compliance|Fully ACID (Atomicity, Consistency, Isolation, Durability)|Often **eventually consistent**; some NoSQL (like MongoDB) now support ACID for limited use|
|Best For|Banking, finance, apps where data **accuracy is critical**|Social media feeds, caching, real-time analytics where **speed matters more than absolute consistency**|

### ‚úÖ Examples of Use Cases

|SQL|NoSQL|
|---|---|
|Banking systems|Social media platforms|
|Inventory & ERP systems|Content management systems|
|E-commerce apps needing complex queries|Real-time analytics (IoT, gaming)|
|Applications needing strong relationships between tables|Big data, caching, recommendation engines|

### ü§ù Key Takeaways

1. **SQL = Structured & Reliable**
    
    - Use when data is structured, relationships matter, and consistency is critical.

2. **NoSQL = Flexible & Scalable**
    
    - Use when data is unstructured, schema evolves, or you need horizontal scalability.

3. **Modern Apps Often Use Both (Polyglot Persistence)**
    
    - Example: A social media app may use SQL for user accounts and NoSQL for posts, feeds, and caching.

## DBMS
**DBMS** stands for **Database Management System**. It‚Äôs **software that lets you create, manage, and interact with databases** efficiently and securely.

Think of it as the **middle layer between the database (data storage) and users/applications**‚Äîit handles all the complexity so you don‚Äôt have to deal with raw data files manually.

### üíº Core Functions of a DBMS

1. **Data Storage, Retrieval & Update**
    
    - Handles storing data on disk, retrieving it, and modifying it safely.

2. **Data Security & Access Control**
    
    - Ensures only authorized users can access or modify data.

3. **Data Integrity & Consistency**
    
    - Enforces rules (like primary keys, foreign keys, constraints) to prevent invalid or inconsistent data.

4. **Backup & Recovery**
    
    - Automatically manages backups and restores data if something goes wrong.

5. **Transaction Management**
    
    - Supports **ACID properties** (Atomicity, Consistency, Isolation, Durability) for safe multi-step operations.

6. **Concurrency Control**
    
    - Handles multiple users accessing or modifying data simultaneously without conflicts.

### üß© Types of DBMS

1. **Relational DBMS (RDBMS)**
    
    - Data stored in **tables with rows & columns**
    
    - Examples: MySQL, PostgreSQL, Oracle
    
    - Uses SQL

2. **NoSQL DBMS**
    
    - Data stored in flexible formats: documents, key-value pairs, graphs, columns
    
    - Examples: MongoDB, Cassandra, Redis
    
    - Often schema-less

3. **Hierarchical DBMS**
    
    - Data stored in a **tree-like structure**
    
    - Example: IBM IMS

4. **Network DBMS**
    
    - Data stored in **graphs with nodes and relationships**
    
    - Example: IDMS

### üí™ Why DBMS Is Important

- Without a DBMS, managing large datasets is chaotic and error-prone.

- DBMS provides **efficiency, security, and consistency**.

- It‚Äôs the backbone of almost every modern software application‚Äîfrom web apps to banking systems.

## Tables
In **MySQL**, a **table** is the **fundamental structure where data is stored**. You can think of it like a spreadsheet: it has **rows** and **columns**, where each row represents a single record, and each column represents a field or attribute of that record.

### Structure of a Table

- **Columns (Fields/schema)**: Define the type of data stored, e.g., `id`, `name`, `email`.

- **Rows (Records)**: Each row stores a single instance of data.

- **Example Table: `users`**
	
|id|name|email|age|

|1|Sankho| sankho@mail.com |21|
|2|Riya| riya@mail.com |22|
	
- Here, `id`, `name`, `email`, `age` are **columns**.
	
- Each row is a **record**.

#### Constraints:
In **MySQL**, **constraints** are rules you apply to table columns to **control the type of data allowed**, ensuring **accuracy, integrity, and consistency**.

 **Types of Constraints in MySQL**:

##### üîπ NOT NULL

- Ensures a column **cannot be empty**.
	
```sql
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL
);
```

‚û°Ô∏è Here, every user **must** have a name.

##### üîπ DEFAULT

- Provides a **default value** if none is specified.
    
```sql
CREATE TABLE orders (
    id INT PRIMARY KEY,
    status VARCHAR(20) DEFAULT 'pending'
);
```

‚û°Ô∏è If no `status` is given, it automatically becomes `'pending'`.

##### üîπ UNIQUE

- Ensures all values in a column are **different** (no duplicates).
    
```sql
CREATE TABLE employees (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE
);
```

‚û°Ô∏è No two employees can have the same email.

##### üîπ PRIMARY KEY

- **Unique + Not Null**, identifies each row in a table.

- Usually applied to `id` columns.
	
```sql
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    name VARCHAR(50)
);
```

‚û°Ô∏è Each student is uniquely identified by `student_id`.

##### üîπ AUTO_INCREMENT

- Automatically generates a **sequential number** for each new row.
    
```sql
CREATE TABLE products (
    product_id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);
```

‚û°Ô∏è New products automatically get IDs: 1, 2, 3‚Ä¶

##### üîπ FOREIGN KEY

- Creates a **relationship between two tables**.

- Ensures the value exists in another table (referential integrity).
    
```sql
CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

‚û°Ô∏è `user_id` in `orders` must exist in the `users` table.

##### üîπ CHECK (MySQL 8.0+)

- Ensures values meet a condition.
    
```sql
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    balance DECIMAL(10,2) CHECK (balance >= 0)
);
```

‚û°Ô∏è Balance cannot be negative.

##### üîπ INDEX (not exactly a constraint, but related)

- Improves speed of queries (not about validity, but about performance).
    


#### Primary Key:
A **Primary Key** is one of the **most important constraints** in MySQL (and relational databases in general). It uniquely identifies every row (record) in a table.

##### üî∂ Key Characteristics

- **Uniqueness** ‚Üí No two rows can have the same primary key value.

- **Not Null** ‚Üí A primary key column cannot have `NULL`.

- **One per table** ‚Üí Each table can have **only one primary key** (but it can be made of multiple columns ‚Üí _composite key_).

- **Indexing** ‚Üí MySQL automatically creates an index on the primary key, making lookups faster.

##### üî∂ Example: Single Column Primary Key

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE
);
```

- `id` is the **primary key**.

- It uniquely identifies each user.

- `AUTO_INCREMENT` makes MySQL assign IDs automatically (1, 2, 3‚Ä¶).

##### üî∂ Example: Composite Primary Key
When uniqueness depends on more than one column.

```sql
CREATE TABLE enrollment (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id)
);
```

- Here, a student can enroll in many courses, and a course can have many students.

- But the combination `(student_id, course_id)` must be unique.

##### üî∂ Why Primary Key is Important

- Prevents **duplicate rows**.

- Acts as a **reference point for foreign keys** in other tables.

- Ensures **data integrity** (no two rows can represent the same entity).

#### Foreign Key:
A **Foreign Key** in MySQL is a column (or set of columns) in one table that **references the Primary Key in another table**. It creates a **link (relationship)** between the two tables and ensures **referential integrity** (no orphaned or invalid references).

##### üî∑ Key Characteristics

- A **foreign key** points to a **primary key** (or unique key) in another table.

- Ensures that values in the child table **must exist in the parent table**.

- Prevents invalid data (e.g., you can‚Äôt assign an order to a non-existent user).

- Supports **cascading actions** (update/delete rules).

##### üî∑ Example: Foreign Key in Action
Let‚Äôs model a simple system: **Users and Orders**.

```sql
-- Parent Table
CREATE TABLE users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL
);

-- Child Table
CREATE TABLE orders (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    product VARCHAR(100),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

How this works:

- Each `order` must be linked to a valid `user`.

- If you try to insert an order with a `user_id` that doesn‚Äôt exist in `users`, MySQL will reject it.

##### üî∑ Cascading Actions
When the parent row changes, what happens to the child rows? You can control this with **ON DELETE** and **ON UPDATE**.

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    product VARCHAR(100),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```

Options:

- **CASCADE** ‚Üí Delete/update child rows automatically when parent is deleted/updated.

- **SET NULL** ‚Üí Set the foreign key column to `NULL` if parent is deleted.

- **RESTRICT / NO ACTION** ‚Üí Prevent deletion/update if child rows exist (default).

#### Difference b/w Primary and Foreign key

```mysql
# Primary Key
	# It is a column (or set of columns) in a table that uniquely identifies 		each row. (a unique id)
	# There is only 1 PK & it should be NOT null.

# Foreign Key
	# A foreign key is a column (or set of columns) in a table that refers to 		the primary key in another table.
	# FKs can have duplicate & null values.
	# There can be multiple FKs.
```

#### ‚úÖ Basic Syntax



‚û°Ô∏è More efficient than inserting one row at a time.

If you provide values for **all columns in exact order**, you can skip the column list.


- `NULL` is used for `AUTO_INCREMENT` column so MySQL assigns the next number.

==‚ö†Ô∏è **Warning:** This is risky if table structure changes. Always better to list columns explicitly.==

#### ‚úçÔ∏è Insert Using Default Values
If a column has a **DEFAULT** constraint, you can skip it.

```sql

INSERT INTO orders (product) VALUES ('Laptop');
```

‚û°Ô∏è Status automatically becomes `'pending'`.

#### Clauses
The `SELECT` statement is used to retrieve data from a database. Clauses are additional keywords that modify or filter the result of the query.

##### 1. FROM

- Defines the table(s) from which data will be fetched.
	
```sql
SELECT column1, column2
FROM table_name;
```

##### 2. WHERE

- Filters rows based on a condition.
	
```mysql
# Where clause : To define some conditions
SELECT name, age FROM userInfo 
WHERE age > 15;
```

==Disclaimer: We can't use aggregate function with Where clause, in those case we have to use subqueries.==

 üìò **Operators in `WHERE` Clause**
The `WHERE` clause is used to filter records based on a condition. Operators define how the filtering is applied.

###### **1. Arithmetic Operators**
Used to check values using mathematics.

- `+` ‚Üí Addition

- `-` ‚Üí Subtraction

- `*` ‚Üí Multiplication

- `/` ‚Üí Division

- `%` ‚Üí Modulo (Remainder)

```mysql
# Arithmetic Operator: +, -, *, /, %
SELECT name,age FROM userInfo 
WHERE age + 1 = 17;
```

###### **2. Comparison Operators**
Used to compare values.

- `=` ‚Üí Equal to

- `!=` or `<>` ‚Üí Not equal to

- `>` ‚Üí Greater than

- `<` ‚Üí Less than

- `>=` ‚Üí Greater than or equal to

- `<=` ‚Üí Less than or equal to

```sql
# Comparison Operator: =, !=, <, >, <=, >=
SELECT name, followers FROM userInfo 
WHERE followers <= 400;
```

###### **3. Logical Operators**

Used to combine multiple conditions.

- `AND` ‚Üí All conditions must be true

- `OR` ‚Üí At least one condition must be true

- `NOT` ‚Üí Negates a condition

- `BETWEEN` ‚Üí Checks if a value lies within a range (inclusive)

- `IN` ‚Üí Checks if a value matches any value in a list

```mysql
# Logical Operator: AND, OR, NOT, IN, BETWEEN
SELECT
	name,
	age,
	followers
FROM
	userInfo
WHERE
	age > 15
	AND follow <300;
SELECT name, age, follow FROM userInfo 
WHERE age > 15 OR follow <300;
SELECT name, age, follow FROM userInfo 
WHERE NOT (age > 15 OR follow <300);

	# Some new keywords
SELECT name, followers FROM userInfo 
WHERE age BETWEEN 14 AND 16;
SELECT name, email FROM userInfo 
WHERE email IN ("bob123@gmail.com","mikeBoss@yam.com", "mymail@yahoo.com");
	#If no match is found then sql will skip that.
```

###### **4. LIKE**
Used for pattern matching (with wildcards).

- `%` ‚Üí Matches zero or more characters

- `_` ‚Üí Matches exactly one character

```mysql
SELECT * FROM employees
WHERE name LIKE 'A%';   # names starting with A
```

###### **5. IS NULL / IS NOT NULL**
Checks if a column has `NULL` value.

```mysql
SELECT * FROM orders
WHERE delivery_date IS NULL;
```

###### **6. ANY / ALL**
Used with subqueries.

- `ANY` ‚Üí True if condition matches **at least one** value in the subquery

- `ALL` ‚Üí True if condition matches **all** values in the subquery

```mysql
SELECT * FROM employees
WHERE salary > ANY (SELECT salary FROM employees WHERE department = 'HR');
```

###### **7. EXISTS**
Returns true if the subquery returns one or more rows.

```mysql
SELECT * FROM customers c
WHERE EXISTS (SELECT * FROM orders o WHERE c.id = o.customer_id);
```

###### **8. Bitwise Operators**

|Operator|Meaning|Typical Use in `WHERE`|
|---|---|---|
|`&`|Bitwise AND|Test if specific bit(s) are set|
|`|`|Bitwise OR|
|`^`|Bitwise XOR|Check if bits differ|
|`~`|Bitwise NOT|Test for bit(s) **not** set (via complement)|
|`<<`|Left shift|Build dynamic masks (`1 << n`)|
|`>>`|Right shift|Inspect higher-order bits|

##### 3. DISTINCT

- Removes duplicate rows from the result.
    
```mysql
SELECT DISTINCT country
FROM customers;
```

##### 4. ORDER BY

- Sorts the result set in ascending (`ASC`, default) or descending (`DESC`) order.
    
```mysql
# Order By Clause: Sort the table in ascending (ASC) or descending(DESC) order
SELECT name, follow, id FROM userInfo 
ORDER BY follow ASC;

SELECT name, follow, id FROM userInfo 
ORDER BY follow DESC;

# If not specified then by default it will sort in ASC order.
SELECT * FROM userInfo 
ORDER BY name;
```

##### 5. GROUP BY

- Groups rows that have the same values in specified columns.

- Often used with aggregate functions (`COUNT`, `SUM`, `AVG`, `MAX`, `MIN`).

```mysql
# Group by Clause: Groups rows that have the same values into summary rows. It collects data from multiple records and groups the result by one or more column.

SELECT age FROM userInfo
GROUP BY age;
```

##### 6. HAVING

- Filters groups created by `GROUP BY`.

- Similar to `WHERE`, but applied **after grouping**.

```mysql
# Having clause: Similar to Where i.e. applies some condition on rows but, it is used when we want to apply any condition after grouping.

# WHERE is for the table, HAVING is for a group
# Grouping is necessary for HAVING

SELECT age, max(followers) FROM userInfo
GROUP BY age
HAVING max(followers) > 400
ORDER BY age;
```

##### 7. LIMIT

- Restricts the number of rows returned.
    
```mysql
SELECT *
FROM products
LIMIT 10;
```

##### 8. OFFSET

- Skips a specific number of rows before returning results (used with `LIMIT`).
    
```mysql
SELECT *
FROM products
LIMIT 10 OFFSET 5;
```

#### Aggregate Functions

Aggregate functions perform calculations on a set of rows and return **a single value**.  
They‚Äôre often used with `GROUP BY` to group rows before applying the function.

**üîë Common Aggregate Functions**

##### 1. COUNT()

- Returns the number of rows (non-NULL values if a column is specified).
    
```mysql
SELECT COUNT(*) AS total_students
FROM students;

SELECT COUNT(age) AS known_ages
FROM students;
```

##### 2. SUM()

- Returns the total sum of a numeric column.
    
```mysql
SELECT SUM(salary) AS total_salary
FROM employees;
```

##### 3. AVG()

- Returns the average value of a numeric column.
    
```mysql
SELECT AVG(marks) AS avg_marks
FROM students;
```

##### 4. MIN()

- Returns the smallest value in a column.
    
```mysql
SELECT MIN(price) AS cheapest_item
FROM products;
```

##### 5. MAX()

- Returns the largest value in a column.
    
```mysql
SELECT MAX(price) AS most_expensive
FROM products;
```

##### 6. GROUP_CONCAT() (MySQL-specific)

- Concatenates values from a group into a single string.
    
```sql
SELECT department, GROUP_CONCAT(name) AS employees
FROM staff
GROUP BY department;
```

##### ‚ö° Usage with GROUP BY
Aggregate functions are often paired with `GROUP BY` to calculate values per group.

```sql
SELECT department, COUNT(*) AS total_emp, AVG(salary) AS avg_salary
FROM employees
GROUP BY department;
```

##### ‚ö†Ô∏è Key Points to Remember

- `WHERE` filters rows **before** aggregation.

- `HAVING` filters rows **after** aggregation.

```sql
-- Only show departments with more than 5 employees
SELECT department, COUNT(*) 
FROM employees
GROUP BY department
HAVING COUNT(*) > 5;
```

- Aggregate functions **ignore NULL values** (except `COUNT(*)`, which counts all rows).

### Trancating Table Data 
The `TRUNCATE` command is a **DDL (Data Definition Language)** statement used to **remove all rows** from a table instantly.  
Unlike `DELETE`, it does not delete rows one by one.

#### üîë Syntax

```sql
TRUNCATE TABLE table_name;
```

#### üîç Key Characteristics

- **Removes all rows** from a table.

- **Resets AUTO_INCREMENT** counter (back to 1).

- Cannot use `WHERE` (all rows are always removed).

- Faster than `DELETE` because it deallocates data pages instead of logging each row deletion.

- Considered a **DDL operation** (internally performs a `DROP` + `CREATE` on the table).

#### ‚úÖ Example

```sql
TRUNCATE TABLE orders;
```

üëâ This removes all rows from `orders` and resets any auto-increment columns.

#### ‚ö†Ô∏è Important Notes

- **Irreversible**: Once truncated, data cannot be recovered (unless backups exist).

- Cannot be used if the table is referenced by a **foreign key constraint**.

- Requires **DROP privileges** (not just DELETE privileges).

- Much faster than `DELETE` for large tables.

#### üîÑ TRUNCATE vs DELETE vs DROP

|Command|Removes Rows|Can Use WHERE|Resets Auto-Increment|Table Structure Remains|Speed|
|---|---|---|---|---|---|
|**DELETE**|Yes (row by row)|‚úÖ Yes|‚ùå No|‚úÖ Yes|Slower|
|**TRUNCATE**|Yes (all rows)|‚ùå No|‚úÖ Yes|‚úÖ Yes|Faster|
|**DROP**|Entire table|‚ùå No|‚ùå N/A|‚ùå No (table removed)|Fastest|
